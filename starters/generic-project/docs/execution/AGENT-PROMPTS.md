# <项目名> · 子代理 Prompt 模板

适合 `Agent` 工具调用的 prompt 模板集合。每模板自包含、可空降一个"零上下文"Claude 执行。

---

## 通用公约（所有 spawn 都要带）

每个子代理 prompt 末尾追加：

```
## 必读前置（零上下文空降须知）
项目根：<绝对路径>
- 先读 `CLAUDE.md`（项目纪律权威源）
- 再读 `EXECUTION.md`（3 分钟了解纪律）
- 再读 `docs/execution/DECISIONS.md`（决策清单，不要违反）
- 再读 `docs/execution/BATCH-XX-*.md`（你负责的批次详情）
- 实施未规划于 docs/0X 的需求时，先看 `docs/execution/INTENT-GRAPH.md`（Phase 3 起）
- docs/0X-*.md 是规格，**默认只读**（编辑例外见决策记录）

硬约束：
<在这里同步 CLAUDE.md 的硬约束清单——必须每条都对应有决策记录>

多 Agent 并行一致性（避免同概念多个名字/风格）：
- 启动前先查现有产出物看相关概念是否已有命名/风格
- 复用已有约定；不要起新名/新风格
- 跨 agent 共享的术语必须先在 DECISIONS.md 出现过；没有就向用户确认
- 合并前与同 Batch 其他 agent 的产出做 diff，发现冲突立即对齐

完成后：
- 跑项目的质量检查全套（具体命令见 `docs/03-quality-contract.md` 或 BATCH 文档）
- 报告本次新增 / 修改的产出物列表 + 所有质量检查的结果
- 如本次实施关联 INTENT-GRAPH 中某条意图，回写 §3 对账表把状态置 `implemented`
```

> **使用方式**：派 agent 时把"通用公约 + 模板特定段 + 当前 Batch 任务清单"拼起来作为 prompt。
> 通用公约的"硬约束"段每次必须从 CLAUDE.md 同步，避免漂移。

## 模板 T1：Scaffold Agent（项目骨架）

**用途**：在空目录里初始化项目骨架。
**子代理类型**：general-purpose
**通常并行度**：1（这是所有后续 Batch 的前置）

```
你负责本项目的骨架 Batch。读 docs/execution/BATCH-XX-*.md，按里面的
产出物清单和质量检查执行。

完工验收命令（必须全绿才算完）：
  <在这里列出具体命令>

遇不确定的信息/工具先用 context7 查；不要依赖训练数据。

[附通用公约……]
```

---

## 模板 T2：Task Executor（单任务执行，可并行 N 实例）

**用途**：在已建好的项目骨架上执行单一任务/模块。
**子代理类型**：general-purpose
**通常并行度**：3-9（取决于任务独立性）

```
你负责本项目的任务 **<TASK_NAME>**（候选：<在 BATCH 文档里列出>）。

读 docs/execution/BATCH-XX-*.md 找到你负责的任务段落，按要求产出：
  <产出物路径>
  <对应质量检查路径，如有>

任务特定硬约束（不在通用公约里的）：
<列出本任务特定的限制，例如：
 - 必须遵守的格式/模板
 - 命名约定 / 文件组织约定
 - 质量检查要求
>

质量检查：
- <检查工具 + 通过标准>

验收：
  <具体质量检查命令>

[附通用公约……]
```

> 适用范围：写作的章节创作、设计稿的页面/组件、研究的章节/数据分析、视频的场次制作等。

---

## 模板 T3：Quality Reviewer（独立质量审查）

**用途**：Batch 完工后做独立质量审查。
**子代理类型**：Explore（只读，不改产出物）

```
你是本项目的质量审查 agent。刚完成了 Batch **<BATCH-ID>**，请独立评估，
报告 < 400 字：

1. 是否遵守了项目的硬约束（逐条检查）
2. 是否遵守了质量契约（对照 docs/03-quality-contract.md）
3. 产出物是否完整（无遗漏）
4. 风格/格式是否一致
5. 本批次 DECISIONS.md 是否有新决策忘记登记
6. 本批次完成后是否回写了 INTENT-GRAPH §3 对账（如适用，Phase 3 起）

输出"⚠️ 风险清单"和"✅ 已通过清单"。**不要自己动手改产出物**。

[附通用公约……]
```

## 模板 T4：Rescue Agent（卡死时）

**用途**：主 Claude 在某个 Batch 卡了 15+ 分钟时派出做独立诊断。
**子代理类型**：general-purpose

```
本项目 Batch <ID> 遇到阻塞。
症状：<具体描述：期望 vs 实际>
已尝试：<列出>
相关产出物：<路径>
相关文档：docs/execution/BATCH-<ID>-*.md

请独立诊断根因并给出最小修复方案。**不要直接改产出物**，输出诊断报告即可。
```

## 模板 T5：Migration / Refactor Agent（已有资产改造）

**用途**：在 Phase 3 / 已有项目中做带风险的改造/迁移。
**子代理类型**：general-purpose
**通常并行度**：1-2（带 review agent）

```
你负责本项目的改造任务：**<TASK_NAME>**。

读 docs/execution/BATCH-XX-*.md + 现状产出物（先摸清范围）。

硬约束：
- 改造前必须有对现状的基线质量检查；如无，**先补检查再动手**
- 每一步必须可独立回滚
- 改动完成后必须跑全套质量检查
- 如果改动范围超出 BATCH 文档预期 → 停下来报告，让用户决定是否扩大范围

验收：
  <全套质量检查命令>

[附通用公约……]
```

## 领域特定模板（按需追加）

以上 T1-T5 是通用模板，覆盖大多数场景。如果你的领域有特殊工作流，
可追加 T6 / T7 ... 并在 DECISIONS.md 记录。

### 领域模板示例

**写作领域**：
- T6：Chapter Writer（章节创作）：负责单章初稿 + 大纲对齐
- T7：Editor Agent（编辑审校）：负责风格一致性 + 情节连续性检查

**设计领域**：
- T6：Component Designer（组件设计）：负责单个组件/页面设计
- T7：Design System Auditor（设计系统审计）：负责组件变体覆盖/无障碍检查

**研究领域**：
- T6：Literature Reviewer（文献综述）：负责某子领域的文献检索与分析
- T7：Data Analyst（数据分析）：负责实验数据统计与可视化

---

## 如何使用本模板

1. 替换所有 `<...>` 占位符为项目实际内容
2. 通用公约的"硬约束"段必须从 CLAUDE.md 同步，不要漂移
3. 派 agent 时把模板 + 当前 Batch 的具体任务拼起来作为 prompt
4. 保留需要的模板（T1-T5），刪掉不需要的
5. 需要领域特定模板时追加 T6+ 并在 DECISIONS.md 记录
