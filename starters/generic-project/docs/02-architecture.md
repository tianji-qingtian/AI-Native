# 项目架构（Phase 1 产出）

> 本文件由 Phase 1 契约骨架阶段产出。这是给后续 Batch 提供稳定地基的契约——
> 一旦定稿，跨 Batch 改动需要立决策记录。

---

## 1. 核心模块图

```
<在这里画项目架构图（ASCII 或 Mermaid）。
 至少要表达：
 - 主要模块 / 阶段
 - 它们之间的依赖/数据流方向
 - 对外交付的产出物/接口
>

例（写作项目）：
 ┌─────────────┐         ┌──────────────┐
 │  大纲管理    │ ──────>│  章节写作     │
 └─────────────┘         └──────┬───────┘
                                │
                ┌───────────────┼───────────────┐
                ▼               ▼               ▼
        ┌──────────────┐ ┌──────────┐ ┌──────────────┐
        │  编辑审校     │ │ 风格检查  │ │  导出发布     │
        └──────────────┘ └──────────┘ └──────────────┘
```

---

## 2. 模块间依赖契约

<在这里写：模块间的依赖规则 + 谁依赖谁。>

例：
- 章节写作依赖大纲管理（必须先有大纲才能开始写）
- 编辑审校依赖章节写作（必须先有初稿才能审校）
- 风格检查可独立运行，不依赖其他模块

---

## 4. 不可逆边界

<列出"现在定了之后改不动 / 改起来代价巨大"的决策。这一节是给未来的自己看的：
告诉自己哪些边界不要轻易碰。>

例：
- 章节格式（所有章节按此格式，事后改动 = 全量迁移）
- 叙述视角/时态（全书一致，中途改动 = 全书重写）

---

## 5. 待澄清

<Phase 1 结束时还没拍板、留到 Phase 2 早期再决定的决策点。
每个待澄清项必须有"什么时候必须决定"的截止 Batch。>

例：
- 目标读者画像细化 → 必须在 B1 之前决
- 是否支持多视角叙述 → 必须在 B3 之前决

---

## 6. 如何使用本模板

1. 每个关键决策必须有决策记录编号，没有就先去写
2. 不可逆边界一定要明确写——避免 6 个月后的自己想不起来
3. 架构图复杂度按项目实际需要——简单项目几行就够，复杂项目可拆多张子图
4. 这份文件 Phase 2-3 期间默认只读
