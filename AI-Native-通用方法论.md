# AI-Native 通用工作流方法论

## 0. 谁该读这份文档

- 用 Claude Code（或同类 AI agent CLI）作为主力工具的创作者/工程师/研究者
- 想把"AI-native 工作流"落到实操而非停留在概念的实践者
- 准备从零启动新项目，或者把 AI agent 引入已有项目/资产的团队

适用领域：软件开发（AI-Native-Code）、小说写作（AI-Native-Write）、设计创作（AI-Native-Design）、学术研究（AI-Native-Research）、视频制作（AI-Native-Video）等任何可由 AI agent 执行子任务的创造性工作。

如果你只是偶尔让 Claude 改点东西，不要读——这套方法论的成本只有在中等规模以上项目才回本。

***

## 1. 核心诊断：什么是 AI-Native，什么不是

AI-Native 工作流的核心主张：用「意图驱动 + Living Spec + 多 Agent 受约束并行」替代传统"人类写完整规格 → 人类分工执行"的瀑布模式。

**对的部分**：

- ✅ 静态规格文档与产出物必然脱节
- ✅ 单 Agent 上下文有限，多 Agent 协作是必然
- ✅ 机器可执行的质量契约应作真相源，不是静态文档
- ✅ 规格应该 Living——能随项目演进而同步

**错的部分**（照搬会撞墙）：

- ❌ "AI 可以自主推导实现路径"——当前 LLM 在长链路推理、跨上下文取舍、识别"自己不知道什么"上仍有显著失败率
- ❌ "意图清晰度是稀缺资源，执行能力不稀缺"——执行能力依然稀缺，只是分布不同；跨多模块的复杂集成 AI 仍会产出"看起来对但语义错误"的结果
- ❌ "传统规格 = 人天估算 = 必须废除"——人天估算是传统规格的副作用，不是本质。规格的本质（"可被审阅的意图载体"）永远需要存在，只是形式可以变
- ❌ 把"AI 自主"放在头部假设——会导致放任，最后产出的结果看起来都过检查，但产品线索糊掉

**结论**：这套范式真正的价值不是"让 AI 自主"，而是**强迫规格双向同步 + 让多 Agent 在共享契约下并行**。"自主"是修辞，"受约束的局部规划"才是事实。

***

## 2. 责任分工：永远不要混淆

在任何阶段、任何场景下，下面这张分工表都成立。**搞错分工是这套方法论失败的最大原因**。

| 工作 | 谁做 | AI 能不能替代 |
|------|------|-------------|
| 商业判断 / 用户访谈 / 同理心 | 人类 | ❌ 永远不能 |
| 顶层意图定义 | 人类 | ❌ AI 可起草，人类必须定稿 |
| 不可逆决策（领域方向 / 核心约束） | 人类 | ❌ AI 可列选项，人类拍板 |
| 收集"使用证据"（数据 / 反馈 / 指标） | 人类（或系统） | ❌ AI 通常无法直接访问 |
| 验收（"AI 报告全绿 ≠ 真的对"） | 人类 | ❌ |
| Batch 任务书起草 | AI | ✅ 可起草，人类审 |
| 单 Batch 内的任务执行 | AI（Agent Team） | ✅ AI 主力 |
| 质量契约守门 | 工具自动化 | ✅ AI 配置 + 自动跑 |
| 跨 Batch 全局一致性 | 质量契约网络 | ⚠️ 不靠 AI 记忆，靠自动化检查 |
| 决策溯源 | DECISIONS.md | ✅ AI 起草，人类审 |
| 规格双向同步 | INTENT-GRAPH §3 对账表 | ✅ AI 回写，人类抽查 |

**关键洞察**：把任何一行"人类做"的事委托给 AI，就是这套方法论失败的开始。

***

## 3. 通用 4-Phase 框架

无论什么领域、无论空项目还是已有资产，工作流都遵循这 4 个阶段：

```
Phase 0  上下文凝练       人类主笔 + AI 反问
   │     空项目：顶层意图（前瞻 ≤ 5 条）
   │     已有项目：现状考古（回顾 + 隐性约束标注）
   ▼
Phase 1  契约骨架         传统规格 + 决策记录
   │     空项目：领域架构 / 工具链 / 质量契约初版
   │     已有项目：反推意图 + 隐性约束显性化
   ▼
Phase 2  质量契约网络     领域特定的自动化验证体系
   │     这是质量保障的真相源，所有后续工作的安全网
   ▼
Phase 3  意图驱动          INTENT-GRAPH + Agent Team
         此时 AI 才能真正发挥规模优势
```

**两个铁律**：

1. **Phase 0/1 不能纯意图驱动**——AI 没有素材可推，强行用意图卡会写出空想
2. **Phase 2 不能跳过**——没有质量契约网络，Phase 3 的 Agent Team 并行会破坏一致性

## 4. 场景 A：空项目（从零启动）

### 4.1 起点

你脑里有个想法，可能伴随：一段口语描述、一份 AI 生成的初稿、几张原型图/参考。

### 4.2 反模式（一开始就别这么干）

| 反模式 | 后果 |
|--------|------|
| "Claude，根据这段描述生成完整规格" | 得到一份充满想象、与真实意图脱节的虚假规格 |
| "Claude，开始干活吧"（跳过 Phase 0） | AI 产出能跑但完全不是你要的东西 |
| Phase 0 就启用 INTENT-GRAPH | 没有产品没有用户，证据触发条件全是空想 |
| Phase 1 就开大量 Agent 并行 | 没有契约网络，多 Agent 命名/风格/抽象一致性必崩 |
| 项目纪律写进 ~/.claude/projects/<hash>/memory/ | 仓库一拷走纪律全丢 |

### 4.3 Phase 0 · 意图凝练（1-3 天）

**输入**：自然语言愿景

**输出**：`docs/01-vision.md`（≤ 5 条顶层意图，每条带证伪条件）

**Claude Code 角色**：反问者 + 起草者

**关键差别**：顶层意图**不是任务清单**，是**可证伪的成功假设**。

| ❌ 任务式 | ✅ 意图式 |
|-----------|-----------|
| "写一本玄幻小说" | "读者能从第一章追读到结局，中途弃书率 < 30%" |
| "做一个品牌 VI 系统" | "新设计师只看 VI 手册就能独立产出风格一致的物料" |
| "拍一支产品宣传片" | "目标受众看完后品牌记忆度提升 ≥ 20%" |
| "调研 LLM 在医疗领域的应用" | "产出的调研报告能被 3 位以上临床医生评为'可直接指导采购决策'" |

**Claude Code 工作流**：

1. 用户用自然语言描述愿景（5-15 句话）
2. Claude 走强制反问：「为什么需要？什么场景下用？什么场景下不用？如果不做用户怎么解决？」
3. 反问 ≥ 1 轮后，Claude 起草 ≤ 5 条顶层意图
4. 用户审 / 改 / 砍 / 合并，定稿

**完成判定**：≤ 5 条顶层意图 + 每条都能用客观指标验证

### 4.4 Phase 1 · 契约骨架（3-7 天，**必须传统范式**）

**为什么不能纯意图驱动**：

- AI 没有任何产出物可读，无法做"扫描发现模式"
- 领域方向/核心约束是不可逆的全局决策，等证据等不起
- 后续 Batch 需要稳定地基

**输出**：

- `CLAUDE.md`（项目纪律权威源；含硬约束、协作模式、工具链、规划范式）
- `EXECUTION.md`（续跑剧本，下次新会话进来读什么）
- `docs/02-architecture.md`（项目架构/工作流架构，**不是 17 份**）
- `docs/03-quality-contract.md`（质量契约初版——领域特定的验证标准与工具）
- `docs/execution/DECISIONS.md`（初始 5-10 条决策记录：工具链选型 / 架构边界 / 不可逆决策）
- `docs/execution/AGENT-PROMPTS.md`（子代理模板，含通用公约）

**Claude Code 角色**：起草者 + 反问者

- "为什么用这个工具而不是那个？写到 DECISIONS.md 里"
- "工作流分几个阶段？每个阶段的产出物是什么？"
- "质量怎么验证？靠什么工具/标准？"

**反模式**：让 AI 自己生成完整规格——它会编。Phase 1 必须是**人类主笔、AI 起草框架、人类填理由**。

**完成判定**：

- [ ] CLAUDE.md 已含项目硬约束（每条都能在某条决策记录找到理由）
- [ ] 项目架构能 1 屏装下，不超过 5 个核心模块
- [ ] DECISIONS.md ≥ 5 条决策记录
- [ ] 任何一个新会话读 CLAUDE.md + EXECUTION.md 能 5 分钟入门

### 4.5 Phase 2 · 核心产出 + 质量契约网络（1-4 周）

**输出**：

- `docs/execution/PROGRESS.md`（Batch 看板，依赖图）
- `docs/execution/BATCH-XX-*.md`（每个 Batch 一份）
- 质量契约配置（与产出物同步建）
- 核心链路产出物

**Batch 拆分原则**：

- ≤ 10 个 Batch
- 每个 Batch 验收的是"哪条顶层意图变得可验证"，**不是**"规格 §3.4 功能列表"
- 严格依赖顺序——下游 Batch 依赖上游产出

**Claude Code 工作流**（每 Batch）：

1. 用户写 `BATCH-XX-*.md` / Claude 起草后用户审
2. 派 Agent Team 执行（具体 agent 类型取决于领域）
3. 完成后回写 `PROGRESS.md` + 必要时追加 `DECISIONS.md`
4. **每个 Batch 末尾人类必须验收**——AI 报告"全绿"≠ 真的对

**质量契约网络与产出物同步建**：

质量契约是**领域特定的**自动化验证体系。不同领域的质量契约示例：

| 领域 | 质量契约示例 |
|------|------------|
| 软件开发 | mypy / pytest / linter / importlinter / CI |
| 小说写作 | 风格一致性检查 / 情节连续性校验 / 字数/章节统计 / AIGC 检测 |
| 设计创作 | 设计系统规则 / 色彩/字体一致性 / 组件变体覆盖 / 无障碍检查 |
| 学术研究 | 引用完整性 / 数据溯源 / 论证逻辑链 / 同行评议清单 |
| 视频制作 | 分镜-成品对账 / 时长约束 / 品牌素材合规 / 音频响度标准 |

**INTENT-GRAPH 此时仍未启用**——它是 Phase 3 工具。

**完成判定**：

- [ ] 顶层意图至少 1 条可验证
- [ ] 质量契约全部通过
- [ ] DECISIONS.md 已记录所有 Batch 内的非平凡决策

### 4.6 Phase 3 · 意图驱动（永远不结束）

**触发条件**：核心链路真跑通，开始有真实使用者/受众。

**此时 INTENT-GRAPH 才发力**：

- 使用过程中冒出"要不要加 X"的零散需求 → 写成意图卡
- 建立证据通道（数据/反馈/指标）
- pending → probing → validated → implemented

**Agent Team 的真正用法**：

- 单意图卡 validated → 派 Agent Team 实施
- 同 Batch 多 agent 并行（一般 3-5 个最佳，上限取决于任务独立性）
- 必须读 AGENT-PROMPTS.md 通用公约

**关键纪律**：

- **核心产出落地前不写意图卡**（会变空想）
- **active 意图 ≤ 12 条**（超出强制合并 / falsify）
- **证伪条件必须可观测**（能查出客观数据）
- **user override 是紧急通道，不是默认**（窗口内比例 > 50% 视为范式失败）

## 5. 场景 B：已有项目/资产

### 5.1 起点

已有产出物在运行/使用中，可能伴随：过时的规格文档、不完整的记录、缺质量检查、有历史约束、多人协作。

**比空项目更复杂的地方**：

- 现有产出物里有大量"未文档化的隐性约束"——AI 容易把"看起来多余的东西"当"待简化目标"，但实际可能是历史原因必需的
- 已有规格可能严重脱节——AI 把过时假设当真，做出实际不对齐的规划
- 团队已有自己的工作流——引入 AI agent 范式有兼容性问题

### 5.2 反模式

| 反模式 | 后果 |
|--------|------|
| "Claude，扫一下自动生成规格" | AI 写出表面准确实际错误的规格（识别不到隐性约束） |
| 一上来开大量 Agent 并行改造 | 没有质量契约的项目会被改崩 |
| 把已有规格直接喂 AI | AI 把过时假设当真 |
| 跳过 Phase 0 直接挂意图卡 | 已有资产无尽 backlog |
| Phase 2 质量契约没建好就让 AI 大改 | 看似对实际崩，且没有回滚锚点 |

### 5.3 Phase 0 · 考古（1-2 周）

**输入**：已有产出物 + 可能过时的规格

**输出**：`docs/execution/PROJECT-MAP.md`（带人类批注的项目反向地图）

**Claude Code 角色**：考古者（**只列事实，不写"为什么"**）

**工作流**：

1. 派 agent 扫描，产出地图：模块结构 / 核心产出物清单 / 依赖关系 / 主要工作流 / 质量现状
2. **人类逐段标注**：哪些是核心？哪些是技术债？哪些是历史约束不能动？哪些是死资产？
3. 把"为什么这段/这个产出物存在"显性化到 DECISIONS.md

**关键风险**：AI 容易把"冗余/过度设计"当"待简化目标"。比如一个复杂流程，看起来该简化，实际可能对应两年前的合规整改。**人类标注是不可省的**。

**完成判定**：

- [ ] PROJECT-MAP.md 覆盖核心模块
- [ ] 每个核心模块有"为什么存在"批注
- [ ] 已识别的隐性约束 ≥ 5 条入 DECISIONS.md

### 5.4 Phase 1 · 反推意图（1 周）

**输出**：`docs/execution/INTENT-GRAPH.md`（≤ 10 条反推意图，**先不启用证据通道**）

**两类关键产出**：

| 类型 | 含义 | 行动 |
|------|------|------|
| **资产有但没意图** | 疑似冗余/死资产/历史遗留 | 写"待证伪意图"——若长期无人维护无人使用 → 删 |
| **意图有但资产弱** | 产品弱点 / 已知问题 | 写普通意图卡，进入证据通道 |

**Claude Code 工作流**：

1. 基于 PROJECT-MAP，AI 起草反推意图清单
2. 用户审：哪些是真意图、哪些是 AI 误读
3. 定稿入 INTENT-GRAPH，但**状态全部 pending**——证据通道等 Phase 2 质量网络建好

### 5.5 Phase 2 · 建质量契约网络（2-6 周，**最不能省**）

**这是已有项目最大的工作量，但跳过会出大事故**。

**核心目标**：把已有资产的隐性约束显性化为可自动验证的契约。

**工作流**：

1. 先配工具后补验证
2. 补核心路径的基线质量检查（并行度 2-3）
3. 起草追溯决策记录——基于历史变更
4. 逐步收紧质量门

**完成判定**：

- [ ] 核心路径有基线质量检查
- [ ] 质量契约全部通过
- [ ] DECISIONS.md ≥ 10 条追溯决策记录

### 5.6 Phase 3 · 增量挂意图（永远不结束）

**触发条件**：Phase 2 质量契约网络建好。

**规则**：

- 所有**新需求 / 改动**强制挂意图卡
- **已有资产维持现状**——直到有人触碰时才反向补意图
- 意图图谱会逐步覆盖项目
- Agent Team 并行度低（2-3 + 互相 review）
- 每次改动派独立 agent 做反向核验
- 每个改动必须报告"动了哪些已有约束"

## 6. 关键差异与共性

### 6.1 差异表

| 维度 | 空项目 | 已有项目 |
|------|--------|----------|
| Phase 0 关键产出 | 顶层意图（前瞻 ≤ 5 条） | 项目考古地图（回顾） |
| Phase 1 关键产出 | 最小契约骨架（冻结一段） | 反推意图清单 + 隐性约束显性化 |
| Phase 2 关键产出 | 核心链路跑通 + 质量契约（同步建） | 质量契约（先于任何 AI 改造） |
| Phase 3 启用 INTENT-GRAPH 时机 | 核心链路通 | 质量契约建好 |
| Agent Team 并行度 | 高（3-5，最高取决于独立性） | 低（2-3 + 互相 review） |
| Claude Code 主角色 | 起草者 + 执行者 | 考古者 + 增量改造者 |
| 主要风险 | 方向选错 | AI 误解隐性约束 |
| 质量契约优先级 | 与产出物同步建 | **先于任何 AI 改造** |

### 6.2 共性原则（永远成立）

| 原则 | 理由 |
|------|------|
| 项目纪律必须入仓库（不入 memory） | 仓库可移植，memory 绑机器 |
| Phase 0/1 不能纯意图驱动 | AI 没素材可推 |
| INTENT-GRAPH 是 Phase 3 工具 | Phase 0 启用会写空想意图 |
| 不可逆决策永远人类拍板 | 不可逆决策不能交给 AI |
| 质量契约网络是真相源（非静态文档） | 静态文档会腐烂，自动化检查不会 |
| Agent Team 并行度看上下文风险 | 风险高 → 并行度低 |
| 用户验收不可省 | "AI 报告全绿"≠"真的对" |
| 决策必须留记录 | 6 个月后没人记得为什么 |

## 7. Agent Team 使用指南

### 7.1 什么时候开 Agent Team

**应该开**：

- 跨模块执行（同一 Batch 要动多个独立模块）
- 可并行的独立工作（不同维度、不同模块）
- 多步骤任务（研究 → 规划 → 执行 → 校验）
- ≥ 2 个独立子任务

**不应该开**：

- 单文件/单任务小改
- 纯查询 / 探索
- Phase 0 / Phase 1（没有可并行的素材）
- Phase 2 质量契约未建好的已有项目

### 7.2 并行度选择

| 项目类型 | 推荐并行度 | 理由 |
|---------|-----------|------|
| 空项目 Phase 2 核心链路 | 3-5 | 有质量网兜底 |
| 空项目 Phase 3 实施意图卡 | 3-5 | 同上 |
| 空项目高度独立子任务 | 5-9 | 任务间零依赖 |
| 已有项目 Phase 2 建质量契约 | 2-3 | 没有兜底 |
| 已有项目 Phase 3 改已有资产 | 1-2 + review agent | 隐性约束风险大 |

### 7.3 多 Agent 并行一致性

并行 agent 协作时最容易出现的问题：同一概念不同命名、风格不一致、边界冲突。**必须**：

- 启动前先查现有产出物看相关概念是否已有命名/风格
- 复用已有约定；不要起新名/新风格
- 跨 agent 共享的术语必须先在 `DECISIONS.md` 出现过；没有就向用户确认
- 合并前与同 Batch 其他 agent 的产出做 diff

***

## 8. 现实预期：加速幅度

**真实加速幅度**（不要吹 10x）：

- 空项目核心产出：30-50% 加速（vs 纯人类）
- 已有项目改造：10-30% 加速（隐性约束风险吃掉部分增益）
- 重复模式（模板化工作、标准化流程）：50-80% 加速
- 创新性工作（方向判断 / 创意突破 / 质量判断）：~0% 加速，AI 是辅助

**任何声称"AI 让你 10x 速度"的都没把 Phase 0/1/2 算进去**——它们只算了 Phase 3 的执行速度。

***

## 9. 反模式速查表

无论空项目还是已有项目都**不要**：

| 反模式 | 后果 |
|--------|------|
| 让 AI 扫一下自动生成完整规格 | 充满想象的虚假规格 |
| Phase 0 启用 INTENT-GRAPH | 空想意图 |
| 跳过决策记录 | 隐性决策必死 |
| 没有质量契约就开 Agent Team 大改 | 看似对实际崩 |
| 把项目纪律写进 memory | 拷贝丢失 |
| 把 user override 当默认通道 | 退化为带额外文档负担的传统模式 |
| 证伪条件用主观词（"用户体验差"） | 永远证伪不掉 |
| 让单一 Agent 跨多 Batch 上下文连续工作 | 上下文爆炸 |
| 不验收就采纳 AI 产出 | "AI 报告全绿"≠"真的对" |

## 10. 领域适配指南

本方法论是**领域无关的骨架**。应用到具体领域时，需要在 Phase 1 明确以下映射：

| 骨架概念 | 软件开发（AI-Native-Code） | 小说写作（AI-Native-Write） | 设计创作（AI-Native-Design） |
|---------|--------------------------|---------------------------|---------------------------|
| 产出物 | 代码 + 测试 | 章节 + 大纲 | 设计稿 + 组件库 |
| 质量契约 | mypy/pytest/linter | 风格检查/情节校验/字数统计 | 设计系统规则/无障碍检查 |
| Agent 类型 | 编码/测试/架构 | 创作/编辑/审校 | 布局/组件/审校 |
| Batch | 功能模块 | 章节/卷 | 页面/组件组 |
| 证据通道 | 埋点/日志/指标 | 读者数据/完读率/弃书点 | 用户测试/设计评审 |

**任何领域都可以套用 4-Phase 框架**，只需要在 Phase 1 定义好该领域的质量契约和 Agent 类型。
